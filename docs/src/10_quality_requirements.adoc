== Requisitos de calidad

=== Árbol de calidad

Basado en las categorías de ISO 25010, con la relevancia observada en
este proyecto:

image::01_2_iso-25010-topics-EN.drawio.png["Categorías de requisitos de calidad (ISO 25010)"]

[cols="1,3", options="header"]
|===
| Categoría (ISO 25010) | Relevancia en AmITraceable

| Seguridad / Privacidad
| **Crítica**. Es el objetivo central del proyecto: consentimiento
  explícito, sin persistencia, minimización de datos.

| Adecuación funcional
| **Alta**. El scoring y las inferencias deben ser explicables y
  trazables (evidencia por atributo).

| Mantenibilidad
| **Alta**. Arquitectura simétrica multiplataforma pensada
  explícitamente para poder añadir plataformas nuevas sin tocar las
  existentes.

| Fiabilidad
| **Alta**. Cobertura de tests significativa (93%+ en el backend),
  quality gate de SonarCloud en cada push.

| Usabilidad
| **Media-alta**. El consentimiento debe quedar claro antes de
  cualquier acción; la selección de plataforma debe ser intuitiva
  (mazo de cartas con arrastre, flechas y puntos, como formas
  redundantes de interacción).

| Portabilidad
| **Media**. Docker Compose para reproducibilidad; documentado el
  manejo de entornos Windows con Python (venvs fuera de carpetas
  sincronizadas por Dropbox, gestión de múltiples versiones de Python).

| Eficiencia de desempeño
| **Baja-media**. No es un sistema de alto volumen (uso académico); los
  límites de extracción (`max_posts`, `max_comments`, `max_media`)
  existen más por minimización de datos que por rendimiento.
|===

=== Escenarios de calidad

[cols="1,3,3", options="header"]
|===
| # | Escenario | Resultado esperado / medido

| QS-1
| Un usuario cierra sesión tras analizar su cuenta.
| No queda ningún rastro de sus datos en el sistema (verificado: no hay
  base de datos; `logout` borra las claves de sesión de esa
  plataforma).

| QS-2
| Se añade una plataforma nueva (ej. X, cuando tenga API disponible).
| Solo requiere: nuevo cliente que devuelva `SocialProfile`, una
  entrada en `_PLATFORM_CLIENT_FACTORIES`, y una entrada en
  `PLATFORM_CARDS` del frontend. No se modifica lógica de otras
  plataformas.

| QS-3
| Falla la API externa de una plataforma (ej. Reddit da 5xx).
| El pipeline lo propaga como error HTTP explícito al frontend
  (`HTTPException`), sin afectar al análisis de otras plataformas ya en
  curso.

| QS-4
| Se ejecuta la suite de tests sin red disponible ni credenciales
  reales.
| Pasa igualmente: todas las integraciones externas están mockeadas
  (`respx`); verificado en CI (runners sin credenciales de
  Reddit/Instagram).

| QS-5
| Un desarrollador nuevo clona el repo en una máquina con una versión
  de Python distinta a la usada en desarrollo.
| Las dependencias resuelven igualmente gracias a los rangos de versión
  en `requirements.txt` (incidente real documentado: Python 3.14 vs
  3.11, ver <<section-technical-risks>>).

| QS-6
| Un atacante descarga el repositorio, modifica el código para quitar
  el aviso de consentimiento, y lo despliega con sus propias
  credenciales.
| No puede acceder a cuentas ajenas: sigue necesitando que el titular
  real de cada cuenta pase por la pantalla de autorización de
  Reddit/Instagram, control fuera del alcance del código modificado
  (ver ADR-5).
|===
