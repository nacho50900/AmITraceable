[[section-deployment-view]]
== Vista de despliegue

=== Infraestructura nivel 1: entornos

[cols="1,2,2", options="header"]
|===
| Entorno | Cómo se levanta | Uso

| Desarrollo local (sin Docker)
| `uvicorn app.main:app --reload --port 3000` (backend, dentro de un
  venv Python) + `npm run dev` (frontend, Vite en puerto 5173)
| Desarrollo día a día; permite hot-reload en ambos lados.

| Local con Docker
| `docker-compose up --build` desde la raíz del repo
| Prueba del sistema completo tal como se desplegaría, incluida
  monitorización.

| Integración continua (CI)
| GitHub Actions, runners `ubuntu-latest`
| Ejecución de tests + análisis SonarCloud en cada push/PR a `main`.

| Publicación de imágenes
| `release-deploy.yml`, disparado al publicar un release en GitHub
| Build y publicación de imágenes Docker de `users` y `webapp` en GHCR
  (`ghcr.io`).
|===

=== Infraestructura nivel 1: despliegue con Docker Compose

[source]
----
+--------------------------------------------------------------------+
|                         Docker host                                 |
|                                                                       |
|  red "monitor-net" (bridge)                                          |
|                                                                       |
|  +-----------+   +-----------+   +--------------+   +-------------+ |
|  |  webapp   |   |   users   |   |  prometheus  |   |   grafana   | |
|  | (nginx)   |   | (uvicorn) |   |              |   |             | |
|  | :80       |   | :3000     |   | :9090        |   | :9091->3000 | |
|  +-----------+   +-----------+   +--------------+   +-------------+ |
|        |               |                |                  |        |
+--------|---------------|----------------|------------------|--------+
         |               |                |                  |
     localhost:80   localhost:3000   localhost:9090     localhost:9091
----

Motivación::
Cada componente en su propio contenedor, en la misma red interna
(`monitor-net`) para que Prometheus pueda hacer *scrape* de
`users:3000/metrics` sin exponerlo públicamente más que a través del
propio backend.

Mapeo de bloques a infraestructura::
[cols="1,2", options="header"]
|===
| Contenedor | Building block(s) que aloja

| `webapp`
| SPA de React compilada (`vite build`), servida como estáticos por
  Nginx. Recibe `VITE_API_URL` como build arg apuntando al backend.

| `users`
| Backend FastAPI completo (todos los módulos descritos en
  <<section-building-block-view>>).

| `prometheus`
| Recolección de métricas expuestas en `users:3000/metrics`
  (`prometheus-fastapi-instrumentator`).

| `grafana`
| Dashboards sobre las métricas anteriores.
|===

=== Infraestructura nivel 2: runner de CI (GitHub Actions)

Motivación::
Reproducir el entorno de tests sin depender de credenciales reales de
Reddit/Instagram: las llamadas a las APIs externas se mockean con
`respx` en los tests, así que el runner no necesita salir a red externa
más que para instalar dependencias y hablar con SonarCloud.

Caché::
`actions/setup-node` y `actions/setup-python` cachean `node_modules`
(vía `package-lock.json`) y la caché HTTP de `pip` (vía
`requirements.txt`/`requirements-dev.txt` como clave), evitando
descargar de cero en cada ejecución si esos ficheros no cambian.
